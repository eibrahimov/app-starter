# Contributing

App Starter improves through careful template changes and lessons from real apps
generated from it. Prefer small, evidence-backed changes that preserve its
one-binary + OpenAPI-source-of-truth identity.

## Setup and validation

See `AGENTS.md` for the full conventions and the step-by-step recipe for adding a resource.

Required before normal PR/commit handoff:

```bash
just lint            # fmt check + clippy -D warnings + tsc, same as CI
just test            # backend black-box tests
just check-typegen   # fail if committed TS types are stale (CI enforces this)
```

Also run when relevant:

```bash
just typegen         # when API/OpenAPI annotations changed; commit schema.d.ts
just build           # release, embedded UI, Docker, frontend build, or packaging changes
just docker-build    # Dockerfile/compose/deployment changes
just desktop-build   # desktop/Tauri sidecar changes
```

If a relevant gate is skipped, explain why in the PR and state the remaining risk.

Use conventional commit messages (`feat:`, `fix:`, `ci:`, `docs:`, `test:`). Keep commits small and focused.

## Approval boundaries

Open an issue or get maintainer approval before changing:

- security posture, authentication, authorization, CORS, request limits, or public exposure defaults;
- dependency policy, major dependency upgrades, Rust edition/toolchain, or JS package manager;
- release workflow, Docker publishing, tags, package names, binary names, or registry targets;
- migration history, data-destructive behavior, or committed migration edits/renames;
- generated-file policy or manual edits to `interface/src/api/schema.d.ts`;
- architectural conventions not represented by both worked examples;
- license policy or copied third-party code.

Routine docs clarifications, tests, bug fixes, and resource additions following
the `items`/`posts` pattern do not need a separate approval issue.

## Hard rules

- Migrations are append-only: never edit or rename a committed file in `migrations/` because sqlx checksums them.
- `interface/src/api/schema.d.ts` is generated by `just typegen` and committed; never hand-edit it.
- JS tooling is bun/bunx only.
- `scripts/setup.sh` is only for fresh-template initialization; do not run it in an initialized generated project.
- Do not remove the Tauri-aware base URL logic in `interface/src/api/client.ts`
  or the permissive CORS default in `src/api/mod.rs` without explicit approval;
  desktop sidecars depend on both.
- Do not copy code from license-incompatible projects or add third-party product
  names. Prefer clean-room reimplementation of patterns over copying files.

## Backporting from generated projects

When you fix something in a generated project that is not domain-specific:

- Template wiring, CI, build, or docs bug: open a PR against the template,
  applying the fix to `items`/`posts` where relevant.
- Adoption friction: open an adoption-friction issue with the command, file, or doc that failed to guide you.
- Reusable pattern: open an issue first describing the problem, recipe location,
  and how it applies to both examples.
- Security, CORS, dependency, release, migration, or architecture defaults: open
  a change request before a PR.

Keep backports free of your app's domain code; port the pattern, not the feature.

See `UPGRADING.md` for upgrade guidance and `docs/contribution-prompts.md` for
copyable issue, change-request, backport, and PR prompts.
